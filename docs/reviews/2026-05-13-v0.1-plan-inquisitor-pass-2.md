### 1. (Pass-1 follow-up) The "58" is right *for the regex it uses*, but the regex is the wrong instrument

**Charge:** The plan's regex (`Issue #\d+|issue #\d+|#\d{3,}|PR #\d+|v5 §|trigger-schema|docs/router`) is narrow. A broader and more honest regex covering `~/.claude|general-purpose|harness|router_lib|dispatch-catalog|dispatch-log` over `src/` returns **259 total hits across the same 5 files** (per my live re-grep). Concrete misses by the narrow pattern: `match_filters.py` calls out `general-purpose` 3 times; `health.py:L11-L13` literally hardcodes `~/.claude/state/router-drift.jsonl` and `dispatch-log.jsonl` in its module docstring; `match.py:L656`, `L670`, `L906` mention `general-purpose` by name; `match.py:L241`, `L257` hardcode `~/.claude/state/...` in code. Many of these are **load-bearing string literals** the implementer must touch when Decision 2 lands, not just docstring chatter.

**Evidence:** Grep counts run 2026-05-13 against `I:\other\claude-wayfinder\.worktrees\port-matcher-issue-2\src`:
- Plan's regex: 58 hits / 5 files (confirms plan).
- `~/.claude|\.claude/` alone: 35 hits / 4 files.
- `harness` alone: 9 hits / 1 file (`health.py`).
- `general-purpose` alone: 6 hits / 2 files (`match.py`, `match_filters.py`).

**Severity:** Major. Decision 5's issue scope says "walk the 58 hits flagged by the grep" — but the implementer will close that issue with 58 hits resolved, while ~200+ other harness-leaks remain. The plan launders a citation-shaped artifact in place of an actual scrub specification.

**Want changed:** Replace the regex with a layered classification: (a) issue-number refs, (b) `~/.claude` hardcoded paths, (c) `general-purpose` / `harness` / `router_lib` identifiers, (d) `v5 §` / `docs/router` cross-references. State counts for each. Decision 2 should reference (b) (the `~/.claude` literals in code, not just defaults), and Decision 5 should own (a), (c), (d). The DoD bullet `grep -r "Issue #\d+\|#\d{3,}..."` (§ 6) likewise needs to widen.

### 3. (New / pass-1 missed) Plan claims a plugin-docs citation that I cannot verify, and the syntax it asserts is unprecedented

**Charge:** § Distribution model § locked says: *"the marketplace name and plugin name are both `claude-wayfinder` — single-plugin repos commonly self-name. Verified syntax per [Discover and install prebuilt plugins](https://code.claude.com/docs/en/discover-plugins) (fetched 2026-05-13)."* The plan presents `/plugin install claude-wayfinder@claude-wayfinder` as canonical syntax. I cannot fetch the URL (`WebFetch` is not in my toolset). ### 4. (New / pass-1 missed) CI in step 1 cannot validate plugin manifests because the manifests don't exist until step 7

**Charge:** Step 1 ships `.github/workflows/ci.yml` (lint + test). Step 7 ships `.claude-plugin/marketplace.json` and `.claude-plugin/plugin.json`. The DoD § 6 last bullet requires `/plugin validate` (or `claude plugin validate`) to pass on the manifests — but that validator is never wired into CI. Six PRs (steps 1-6) merge with no plugin-manifest validation, and the existing `.claude-plugin/plugin.json` (already committed in this worktree — see `git ls-files`) is **already invalid against the v0.1 target schema**: it omits `version`, `homepage`, `repository`, `license`, `keywords` listed in § Distribution model.

**Evidence:** Existing `.claude-plugin/plugin.json` (4-line file) committed on the port branch, missing every v0.1 field except `name`, `description`, `author`. § 4 step ordering puts manifests at step 7. § 6 DoD references plugin validation but the workflow YAML in step 1 has no plugin-validate job.

**Severity:** Major. Step 7 ships the user-facing artifact; if step-1 CI never validates it, the first time anyone confirms the manifest works is the release-day install attempt. This is exactly the failure mode "land CI first" was supposed to prevent (pass-1 charge 7).

**Want changed:** Either (a) step 1's CI workflow includes a `claude plugin validate` job from day one (idempotent — passes against the current partial manifest because `name` is the only required field), or (b) step 7 amends `ci.yml` to add the validation job as part of the same PR that lands the manifests. The current plan describes neither.

### 5. (New / pass-1 missed) `tests/test_skill_dispatch.py` is a deliverable shaped like a test that cannot exist as a test

**Charge:** Decision 3 § Issue to file says: *"add `tests/test_skill_dispatch.py` that exercises the matcher call the skill makes (without needing Claude Code itself — the skill body is markdown but the engine call is testable)."* A skill is a markdown file; the "matcher call the skill makes" only exists when Claude Code (the harness) reads the markdown, decides to invoke the matcher, and orchestrates the subprocess. There is no test fixture for "Claude Code interprets a SKILL.md and dispatches." The only thing this test file can actually exercise is the *Python entry point* the skill subprocess-calls — which is already tested by `tests/test_cli_demo.py`. So `test_skill_dispatch.py` either (a) duplicates the CLI demo test under a misleading name, or (b) requires a Claude Code harness fixture that the plan does not specify.

**Evidence:** Decision 3 § Issue to file paragraph; § 2 file structure listing `test_skill_dispatch.py` as NEW with no scope; no fixture or harness-simulation pattern described anywhere in the plan.

**Severity:** Major. v0.1 ships a deliverable with no specification. Either the test is redundant (drop it) or it's a much bigger task than the plan implies (specify it).

**Want changed:** Decision 3 must specify what `test_skill_dispatch.py` asserts. If it's "the demo CLI is invokable from a subprocess matching the skill's invocation pattern," say so and explain why it isn't `test_cli_demo.py` already. If it's something else, name the something else.

**Want changed:** Either (a) a CI check that asserts `pyproject.toml` version == `plugin.json` version, or (b) Decision 8 explicitly designates one as source of truth and the other as derived (with a release-time script to sync). Plan currently does neither.

## Verdict

Plan v3 is **substantively better than v2** — pass 1's 14 charges drove real improvements, the distribution-shape pivot is well-grounded in the #5 spike, and the consequence statements pass 1 demanded now sit in the right places. But the revision introduced its own failure modes: the doc-scrub regex (Charge 1) saturates the original concern without resolving it; the plugin syntax (Charge 3) and reserved-name check (Charge 11) rest on an Anthropic doc citation I cannot verify; the "blocking preflight" (Charge 8) was renamed but not done; the `test_skill_dispatch.py` deliverable (Charge 5) is unspecified; and the YAGNI citation (Charge 9) was rewritten in a way that satisfies the regex but not the principle. CI sequenced first (Charge 4) reintroduces a different cycle because plugin-manifest validation lands six steps later than CI.

**Not ready to greenlight as-is.** The blocking charges are 1, 3, 4, 5, 8, 9 — the rest can be cleanup at implementation time. Specifically: (1) rewrite the doc-scrub specification with a layered classification or it ships incomplete; (3) verify the plugin sideload syntax against a *live install* before step 7, not after; (4) put `claude plugin validate` in CI from day one or fold it into the step-7 PR; (5) specify what `test_skill_dispatch.py` actually tests or drop it; (8) itemize the 13-test preflight at plan time, not implementer time; (9) ground the YAGNI principle in a citation that supports it, or drop the appeal. Once those are addressed, the plan is ready for implementation.
