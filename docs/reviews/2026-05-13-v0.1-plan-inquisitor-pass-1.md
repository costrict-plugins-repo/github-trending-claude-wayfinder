**Evidence:** Plan § 1.3 item 2 cites "Count: `grep \"Issue #\\d+|issue #\\d+|#\\d{3,}|PR #\\d+|v5 §|trigger-schema|docs/router\"` over `src/`." **What I'd want to see changed:** Re-run the grep at revision time, paste the exact command + count, and update Decision 5's issue scope. If the pattern itself was different at plan-authoring time, paste the older pattern so the delta is auditable.

### 3. Default-path policy plan misses the `CLAUDE_HOME` fallback

**Charge:** Decision 2 enumerates "the `~/.claude/state/...` fallback" as the thing being removed, and the issue scope names `_resolve_catalog_path` and `_resolve_log_path`. But the actual code in `match.py:L223-L257` has a **three-step** lookup — env var → `$CLAUDE_HOME/state/...` → `~/.claude/state/...`. The plan never mentions the `CLAUDE_HOME` middle step.

**Evidence:** `match.py:L235-L241`:
```python
explicit = os.environ.get("DISPATCH_CATALOG_PATH")
if explicit: return Path(explicit)
claude_home_env = os.environ.get("CLAUDE_HOME")
if claude_home_env: return Path(claude_home_env) / "state" / "dispatch-catalog.json"
return Path.home() / ".claude" / "state" / "dispatch-catalog.json"
```

**Severity:** Major. The implementer will read Decision 2, remove the `Path.home()` branch, and either leave `CLAUDE_HOME` intact silently (inconsistent with "no defaults") or remove it without realizing it was a documented contract. Either outcome ships a half-baked policy.

**What I'd want to see changed:** Decision 2 must explicitly state whether `CLAUDE_HOME` stays, goes, or is renamed. If it stays, it should appear alongside `DISPATCH_CATALOG_PATH`/`DISPATCH_LOG_PATH` in the "env-var overrides retained" sentence; if it goes, say so.

### 5. The plan is dishonest about the gap between "Pattern A" and the user's adoption bar

**Charge:** The spike doc § Final answer states bluntly: *"The user's bar is 'marketplace install → works.' Strictly, no skill-subprocess kernel meets it in any language under the current Claude Code model."* The plan codifies Pattern A but never restates this — instead it frames Pattern A as the **convention** (mirroring `microsoft-docs`) without acknowledging that v0.1 **does not meet the user's stated adoption bar**.

**Evidence:** Compare spike doc lines 220-225 (the explicit "the bar is not met" statement) with plan § Distribution model lines 26-31 (which list "implications" but never name the unmet bar). Plan § 5 item 7 says "PyPI unblocks zero new users in v0.1" — true, but conflates the PyPI question with the runtime-bundling question.

**Severity:** Major. CLAUDE.md § Verify Contradictions is explicit: when the public artifact and the prior research conflict, surface and reconcile. The plan finesses the reconciliation.

**What I'd want to see changed:** Add one sentence to § Distribution model: "Consequence: v0.1 does not meet the originally stated 'marketplace install → works' bar. That bar is structurally unreachable today (per #30465) and is the entire purpose of v0.2 spike #6." Make it impossible to read the plan without confronting this.

### 6. The "skipped tests" rewrite is treated as the largest chunk but the planning depth is the shallowest

**Charge:** Decision 6 calls the live-catalog rewrite "the largest single chunk of v0.1 work" and the user explicitly flagged it as the riskiest line item — yet the issue scope is two lines: *"rewrite `_build_live_catalog` to take `agents_dir` + `skills_dir` arguments pointing at the synthetic fixture tree."* There is no validation that the existing synthetic fixtures (`tests/fixtures/agents/code-writer.md`, `tests/fixtures/skills/csv-utils/SKILL.md`) actually exercise the **branches** the 13 cascaded test bodies care about.

**Evidence:** Plan § 1.3 item 4 says ~13 test bodies cascade from `_build_live_catalog`. Decision 6 § Citation lists `tests/fixtures/agents/code-writer.md` and `tests/fixtures/skills/csv-utils/SKILL.md` as "existing synthetic fixtures usable for the rewrite" — singular fixture per category. The cascaded test bodies presumably depend on a multi-agent / multi-skill catalog to exercise scoring tiebreakers, `delegate` vs `ambiguous` gap thresholds, and the routability predicate's plugin-exclusion paths. **Nothing in the plan demonstrates the synthetic-fixture tree is rich enough to do this.**

**Severity:** Major. This is "the largest single chunk of v0.1 work" being scoped on faith. If the fixtures turn out to be insufficient, the issue blows up from "rewrite test plumbing" into "design a synthetic harness corpus" — a substantively bigger task.

**What I'd want to see changed:** Before this issue is filed, run a preflight: enumerate the 13 cascaded tests, classify what each asserts, and verify (or list the gaps in) the existing fixture tree. The preflight result attaches to the issue. If gaps exist, file a sub-issue for fixture authoring before the rewrite issue runs.

### 7. Sequencing claims merge dependency-driven order with surface convenience

**Charge:** § 4 "Suggested merge order" puts #6B (port design docs) first and #5 (CI workflow) seventh "so the workflow runs against the cleaned-up code" — but #5 is the gate that makes the rest of the work auditable. Running CI **late** means six PRs land without lint or build verification.

**Evidence:** § 4 ordinal list, items 1 and 7. No rationale is offered for why catching a `ruff` violation on PR #2 (#6A docstring scrub) is acceptable when CI exists to prevent exactly that.

**Severity:** Major on dev-loop efficiency, Minor on correctness.

**What I'd want to see changed:** Move #5 (CI) to first or second slot. Run a minimum-viable workflow (just `pytest` + `ruff check`) against the port as-is, then layer in lint-strictness as later PRs land. Catching breakage on PR 1 of 8 is the entire reason CI exists.

### 8. PyPI deferral rationale is convenient, not principled

**Charge:** § 5 item 7 says *"PyPI publish → defer to v0.2. PyPI unblocks zero new users in v0.1."* This is wrong for the realistic external evaluator. `pip install claude-wayfinder` is materially lower-friction than "clone the repo, `cd`, `uv pip install -e .`" — especially for an evaluator who isn't already a Python dev.

**Evidence:** Decision 7 quickstart shows `git clone … && cd … && uv pip install -e .` as the install path. Compare against `microsoft-docs` (cited as the precedent) — it is on the marketplace, not on PyPI, but the analog fails: marketplace install is structurally different from PyPI install, and the v0.1 plan cannot use the marketplace path either (per spike #5).

**Severity:** Major. The plan claims Pattern A "matches the precedent every runtime-bearing marketplace plugin sets" — but `microsoft-docs` is reachable via `/plugin install`; v0.1 is reachable only via `git clone`. The precedent is broken at exactly the user-friction step the deferral is supposed to defend.

**What I'd want to see changed:** Either (a) commit to PyPI in v0.1 (small scope: register the name, configure trusted publishing, add the publish job; ~half a day of work, much smaller than the test rewrite), or (b) be honest in § Distribution: "v0.1 has higher install friction than any other Pattern A precedent because we ship neither via marketplace nor PyPI. This is a known gap; the realistic v0.1 audience is people willing to `git clone`."

### 12. Citation form is mixed; some load-bearing claims cite memory or "user brief" rather than verifiable sources

**Charge:** § 1.1 admits *"~10.7k LOC per the user's brief — unverified line count."* § 7 closes the citation discipline section by noting "the only `unverified:` in this plan is the ~10.7k LOC figure." But there are at least two others not marked: (a) § 1.4 references "user-memory feedback" with a parenthetical hedge — that is a memory citation, which CLAUDE.md § Cite Sources explicitly disallows; (b) § 1.3 item 2's "58 occurrences" was unverified at plan-authoring time (charge #1).

**Evidence:** § 1.4 first bullet text; § 1.3 item 2 count.

**Severity:** Major. CLAUDE.md § Cite Sources is explicit: "Memory file paths are **not** an acceptable citation form." The plan attempts to slip by with a parenthetical disclaimer ("Memory paths are not cite-able per CLAUDE.md…; this is here as operating principle, not a load-bearing claim") — but the YAGNI / over-engineering caution **is** load-bearing: it's the criterion driving the cut-line scope decisions throughout § 3 and § Decision 9.

**What I'd want to see changed:** Either (a) cite the YAGNI principle from a non-memory source (CLAUDE.md itself doesn't enshrine it, but the plan-authoring agent's instructions do — cite those instructions by file path + line range), or (b) drop the appeal to memory entirely and let the cut-line decisions stand on their own merits without invoking the principle.

**Charge:** Decision 3 commits the demo to cover four decisions: `delegate`, `self_handle`, `ambiguous`, `needs_more_detail`. But the matcher's enum (`README.md:L14`) lists **seven** decisions: `delegate`, `self_handle`, `self_handle_unaided`, `advisory`, `ambiguous`, `ask_user`, `needs_more_detail`. The plan picks 4 of 7 and calls that "all four decision branches" without explaining what happened to the other three.

**Evidence:** README L14 vs plan Decision 3 item 2.

**Severity:** Major. The README pitch is the "**Seven-decision** enum." The demo coming up short of that pitch makes the demo subtly misleading — exactly what § 1.2 ("rewrite the README to not promise it") is supposed to be fixing.

**What I'd want to see changed:** Either (a) demo covers 7 of 7 (likely the right answer — this is the headline feature; the dev cost is marginal once 4 are wired), or (b) Decision 3 explicitly names the three deferred decisions and Decision 7 README rewrite owns explaining why the demo shows 4 of 7. "All four" without naming what was cut is the worst option.

## Summary

This plan is structurally sound — the nine decisions cover the right surface area, the issue list is well-scoped at the *title* level, and the spike-doc grounding for Pattern A is the strongest section. But the load-bearing facts inside the plan are not rigorously verified (charges 1, 2, 3, 6), the README/plan/code triangle has unsurfaced contradictions (charges 4, 5, 9, 13), and the sequencing + scope decisions hide convenience as principle (charges 7, 8). The pass-2 revision should re-verify every numeric citation against the live code, surface the consequence statements the current plan finesses, and tighten the "largest chunk" scope (charge 6) with a preflight rather than launching it on optimism.
