# v0.2 design — integration flow

- **Drafted:** 2026-05-14
- **Approval:** pending

---

## Context

v0.1 shipped claude-wayfinder as an evaluation surface: the `/dispatch` Skill runs the deterministic 7-decision matcher against bundled fixtures so consumers can inspect the decision logic, but it does not intercept real session traffic (see `skills/dispatch/SKILL.md:L1-L13`). The underlying architecture — scorer, decision ladder, catalog schema — is stable and documented in `docs/schema.md` (algorithm and contract) and `docs/design.md` (rationale).

Issue #37 captures the gap: a consumer who wants to use claude-wayfinder as their actual router has no documented, supported path to wire their own catalog and invoke the matcher from their router agent. Three option shapes (reference router agent, UserPromptSubmit hook, Skill with proactive triggers) were evaluated during a design brainstorm. Approach 1 — Skill-primary, deliberate invocation, no shipped router agent — was chosen for the power-user-with-existing-router audience that is the realistic early adopter. This spec records those settled decisions and defines the four deliverables that constitute v0.2.

The underlying architecture (`docs/schema.md §4`, `docs/design.md`) remains unchanged. v0.2 promotes the existing `build_catalog.py` CLI to a first-class module subcommand and documents the integration contract that was previously informal.

---

## Audience and scope

**Target consumer:** a power-user who already has a working router agent. They author their own router prompt, construct dispatch context themselves (file paths, tool mentions, agent mentions, command prefix), and want a stable, callable matcher plus a way to build a catalog from their own agents and skills. They do not want a shipped reference router agent they have to adopt whole, and they do not want an auto-firing hook that processes raw prompts.

**Explicit non-goals for v0.2:**

- No reference router agent shipped in the plugin — the consumer owns their router.
- No `UserPromptSubmit` hook for automatic dispatch-context extraction from raw prompts — **deferred to v0.3 pending real adoption signal.** The hook primitive is separable from NLP-based dispatch-context extraction; a structured-signal hook (raw prompt + regex-matched paths + slash-prefix, leaving enrichment to the router) is a legitimate v0.3 option. v0.2 defers it because (a) the power-user audience constructs context themselves so the hook adds no value for them, and (b) shipping a hook well requires telemetry on dispatch decision quality that v0.2 does not yet produce.
- No bundled standalone binary (#6 — deferred; Python prerequisite is the v0.2 assumption).
- No marketplace listing on the official Anthropic marketplace (no supported submission flow yet).

## Architecture

Three layers wire together cleanly:

```
Consumer's router agent
        |
        | constructs dispatch context (JSON)
        v
  /dispatch Skill  ──── detects catalog presence ────► demo mode (bundled fixtures)
        |                                               [no catalog configured]
        | real-catalog mode
        | ($DISPATCH_CATALOG_PATH set)
        v
python -m claude_wayfinder.match  (stdin: dispatch context JSON)
        |
        | reads
        v
  dispatch-catalog.json  ◄── python -m claude_wayfinder catalog build
  (consumer's catalog)          (one-time + on refresh)
        |
        v
  decision JSON  (stdout)
        |
        v
Consumer's router agent acts on decision
```

The Python module (`python -m claude_wayfinder.match`) is the canonical entry point, unchanged from v0.1. The Skill is a thin wrapper that manages mode detection and surfaces the decision to the router. The catalog build subcommand promotes the existing `build_catalog.py` CLI to a first-class entry point — the implementation requires adding a `catalog` subparser and a `build` sub-subparser to `cli.py`, then wiring them to delegate to `build_catalog.py`'s existing `main()`.

## Deliverable 1 — Catalog generation surface

**The work is wiring, not redesign.** `build_catalog.py` already has a complete argparse CLI (`build_catalog.py:L1952-L2015`). v0.2's job is to add a `catalog build` subparser to `cli.py`'s `main()`, wire it to call `build_catalog.py`'s existing `main` with the parsed args, and expose `python -m claude_wayfinder catalog build` as a first-class entry point. The implementation requires: (a) add a `catalog` subparser and a `build` sub-subparser to `_build_parser()` in `cli.py`; (b) wire the sub-subparser to call `build_catalog.main()` with parsed args; (c) preserve the existing `demo` subcommand without change. The full parameter surface of `build_catalog.py` is exposed without alteration. No new CLI design is required.

### CLI surface — full parameter table

`python -m claude_wayfinder catalog build` exposes every parameter the underlying `build_catalog.py` CLI exposes. The table below is derived verbatim from `build_catalog.py:L1952-L2015`:

| Flag | Required | Default | Description |
|---|---|---|---|
| `--skills-dir` | **Required** | — | Directory containing skill `SKILL.md` files |
| `--agents-dir` | **Required** | — | Directory containing agent frontmatter `.md` files |
| `--out` | **Required** | — | Output path for `dispatch-catalog.json` |
| `--log` | **Required** | — | Output path for the catalog-generation log |
| `--plugin-overrides-dir` | Optional | `None` | Directory containing plugin-override trigger `.yml` files |
| `--plugins-dir` | Optional | `None` | Directory containing `installed_plugins.json` (Pass 2.5 plugin discovery) |
| `--builtin-agents-dir` | Optional | `None` | Directory containing builtin-agent sidecar `.yml` files (Pass 2.6 builtin discovery) |
| `--corpus` | Optional | `None` | Path to `routing-corpus.jsonl` for corpus-alignment scoring |
| `--project-root` | Optional | auto-detected | Path to the project git root for project-local overlay; when omitted, auto-detected via `git rev-parse --show-toplevel` |

**v0.2 honors issue #10's decision to require explicit paths. Neither `--out` nor `--log` has a default; both must be supplied.** The underlying `build_catalog.py:L1995-2004` marks both as `required=True` with no default. The v0.2 subcommand wrapper passes arguments through directly — no default injection. The integration guide (`docs/integration.md`) will show consumers how to wire these into pre-commit hooks, CI jobs, and shell aliases.

### Console-script entry points

`pyproject.toml`'s `[project.scripts]` table currently declares:

- `claude-wayfinder = "claude_wayfinder.cli:main"` — already declared; the catalog build entry point and entry point for any future subcommands

`claude-wayfinder-match` is **not yet declared** in `pyproject.toml`. Deliverable 1 must add it:

- `claude-wayfinder-match = "claude_wayfinder.match:main"` — direct matcher invocation, equivalent to `python -m claude_wayfinder.match`

Both `python -m` invocation forms remain supported as equivalent invocations. The console scripts are the recommended form for Makefiles, CI jobs, and shell aliases. The integration guide must use console-script forms in all examples.

### Source-tagged precedence model — complete picture

`build_catalog.py` assembles the catalog across six ordered passes (see `build_catalog.py:L1663-L1883`). Each entry carries a `source` tag that records its origin:

- `source="owned"` — user-global skill and agent entries from `--skills-dir` and `--agents-dir` (Passes 1 and 2; `build_catalog.py:L1666`, `L1675`). These are the lowest-precedence entries and cannot be tombstoned or overridden by plugin-override passes.
- `source="plugin"` — dormant entries from plugin-installed skills and agents, discovered via `--plugins-dir` (Pass 2.5; `build_catalog.py:L1520`). Dormant means zero triggers until activated by a plugin-override.
- `source="builtin"` — Claude Code's embedded agents discovered via `--builtin-agents-dir` sidecars (Pass 2.6; `build_catalog.py:L1599`).
- `source="plugin-override"` — entries from plugin-override sidecar files in `--plugin-overrides-dir` that replace or augment plugin-discovered entries (Pass 3; `build_catalog.py:L1460`). Plugin-override entries cannot supersede `source="owned"` entries; the attempt is rejected with a warning (`build_catalog.py:L1811-L1820`).
- `source="project"` — entries from `<project_root>/.claude/skills/**` and `<project_root>/.claude/agents/*.md` when `--project-root` is supplied (Pass 4; `build_catalog.py:L1853`, `L1865`). Project entries override user-global (`owned`) entries on name collision with a logged warning.

Effective precedence ladder (highest to lowest): `project > plugin-override > plugin/builtin > owned`. On name collision, the higher-precedence entry wins; `owned` entries are protected from plugin-override tombstones.

The spec previously described only `project > user` (two sources). That framing was incomplete. All five source tags are part of the v0.2 contract.

### Error handling

| Condition | Behavior |
|---|---|
| Source directory missing or unreadable | Warn to stderr, skip that source, continue building from remaining sources |
| Valid source directory is empty | Silently skip (zero entries contributed from that source) |
| Duplicate name across sources | Source-tagged precedence applies (see above); emit a warning to stderr naming the winning and losing source |
| Output directory does not exist | Create it before writing; fail with a clear error if creation fails |

### Duplicate-name handling — open implementation question

Verify that `build_catalog.py`'s current collision behavior matches the precedence model described above before wiring the subcommand. If the current behavior differs, note the deviation in the implementation PR and align before shipping.

## Deliverable 2 — Skill behavior switch

The `/dispatch` Skill becomes mode-aware. The mode is implicit — detected by the presence of a catalog — not a flag the invoker passes.

### Section 2.1 — Mode detection

**Key principle: demo mode is opt-in by absence of `$DISPATCH_CATALOG_PATH` — it is never a silent failure fallback.** Any real-catalog-mode failure is surfaced to the user. The Skill never silently routes against fixtures when the consumer intended real routing.

**The Skill does not probe the filesystem for a default catalog path.** `match.py:L232-L270` (`_resolve_catalog_path`) follows a strict two-step resolution: explicit argument → `$DISPATCH_CATALOG_PATH` env var → fail loud with `[CATALOG ERROR]`. There is no `Path.home()` default (removed by issue #10; see `match.py:L938`). The Skill honors this contract exactly.

The mode-detection algorithm is a single binary decision:

| `$DISPATCH_CATALOG_PATH` set? | If set, env path readable? | Outcome |
|---|---|---|
| Yes | Yes (readable + valid JSON + valid schema) | **Real-catalog mode** — pass env path through to `python -m claude_wayfinder.match` |
| Yes | No (missing, unreadable, or schema invalid) | **Hard error** — surface `[CATALOG ERROR]`; do NOT fall back to demo mode |
| No | n/a | **Demo mode** — bundled fixtures with a clear "no catalog configured — running in demo mode" banner |

Demo mode activates only when `$DISPATCH_CATALOG_PATH` is absent. When the env var is set but the path is broken, the Skill propagates the hard error — it does not silently fall back to demo.

The `[CATALOG ERROR]` surface behavior is already implemented in `match.py:L866-878`. The Skill must propagate that exit signal to the user rather than swallowing it.

**Stale-mtime behavior:** When the resolved catalog's mtime is older than any source agent/skill file (and sources are enumerable), the Skill emits a warning to stderr — e.g., "Catalog mtime older than source files; consider running `claude-wayfinder catalog build`. Proceeding with stale catalog." — but **proceeds with the stale catalog**. Staleness is a degraded-quality signal, not an outright error.

### Section 2.2 — Real-catalog mode

When activated, the Skill expects the invoker (the consumer's router agent) to provide a composed dispatch context in the same JSON shape as v0.1's matcher input:

```json
{
  "task_description": "...",
  "file_paths": ["..."],
  "agent_mentions": ["..."],
  "tool_mentions": ["..."],
  "command_prefix": "..."
}
```

The Skill runs `python -m claude_wayfinder.match` with this context against the real catalog and returns the matcher's decision JSON verbatim. The consumer's router agent reads the decision JSON and acts on it.

### Section 2.3 — Demo mode

When no catalog is configured, the Skill runs against bundled fixtures. This is the current v0.1 behavior, unchanged. Demo mode preserves the evaluation-surface use case for new consumers inspecting the matcher for the first time.

### Section 2.4 — SKILL.md preamble requirements

The updated `skills/dispatch/SKILL.md` must include a clear preamble that:

1. Names both modes and states the activation condition for each.
2. States that the mode is implicit (presence of `$DISPATCH_CATALOG_PATH`), not a caller-supplied flag.
3. Describes the dispatch context JSON shape expected in real-catalog mode.
4. States that the consumer's router agent must include `Skill` in its `tools:` frontmatter for `/dispatch` to be invocable. Example of correct frontmatter: `tools: Glob, Grep, Read, Edit, Write, Bash, Skill, ToolSearch` — as used in `agents/code-writer.md` in the user's harness.
5. Points to `docs/integration.md` for the end-to-end wiring guide.

The `triggers:` block stays as `command_prefixes: [/dispatch]`. No proactive natural-language firing in v0.2 — the Skill is invoked deliberately by the consumer's router prompt. This was explicitly evaluated (option C in #37) and rejected to keep the integration contract intentional and auditable.

## Acceptance criteria

Derived from #37, adjusted for Approach 1's shape:

- [ ] **Catalog generation CLI is shipped and documented.** `python -m claude_wayfinder catalog build` (and `claude-wayfinder catalog build`) delegates to `build_catalog.py`'s full parameter surface, applying the source-tagged precedence model (all five source tags; `project` highest). `--out` and `--log` are required with no defaults, honoring issue #10. Warns on missing sources and name collisions.
- [ ] **Skill mode detection works end-to-end.** With `$DISPATCH_CATALOG_PATH` set to a valid catalog, `/dispatch` invokes the real matcher against the consumer's catalog and returns the decision JSON. Without `$DISPATCH_CATALOG_PATH` set, it runs demo mode with a clear "no catalog configured" banner. Any real-catalog-mode failure (missing file, invalid JSON, schema error) is surfaced as a hard error — no silent demo fallback. Mode detection follows the single table in Section 2.1 exactly.
- [ ] **Integration guide is published.** `docs/integration.md` contains all five sections defined in Deliverable 3 and is accurate against the shipped CLI and Skill behavior.
- [ ] **Schema documentation is published.** `docs/schema.md` exists and documents the catalog entry schema (including all five `source` tags), the dispatch context schema (5 input fields), and the decision output schema (7 decision types + fields). `docs/schema.md` is the declared stability contract for v0.2 consumers.
- [ ] **README install promise is updated.** The README no longer calls the plugin "evaluation surface only." It describes the v0.2 install as "router scaffold + matcher" and notes that the consumer's router agent must invoke `/dispatch` deliberately.
- [ ] **Graceful degradation is visible.** The mode-detection table in Section 2.1 is implemented exactly: hard error on real-catalog-path failures; demo mode only when `$DISPATCH_CATALOG_PATH` is absent. Stale-catalog warnings are emitted to stderr without blocking routing.
- [ ] **CI coverage and manual verification.** Two halves: (a) *CI-covered* — subprocess tests on `python -m claude_wayfinder catalog build` (input → output catalog shape) and `python -m claude_wayfinder.match` (catalog + context → decision shape) verify the module contract; (b) *Out-of-band verification* — the Skill's mode-detection logic runs inside Claude Code's skill-invocation flow and is not reachable from pytest. Before v0.2 release, the implementer runs a **manual verification protocol** in a real Claude Code session: (1) clear `$DISPATCH_CATALOG_PATH`, invoke `/dispatch` — expect demo-mode banner; (2) set `$DISPATCH_CATALOG_PATH` to a valid file, invoke — expect real-catalog mode; (3) set env var to a missing file, invoke — expect hard error. Results are documented in `docs/release-checklist.md` before release. The Skill behavior switch carries acknowledged **manual-verification risk**.
- [ ] **Console-script entry points declared.** `claude-wayfinder` and `claude-wayfinder-match` are declared in `pyproject.toml`'s `[project.scripts]` table; both resolve correctly after `pip install -e .` and execute identically to their `python -m` equivalents. (`claude-wayfinder` is already declared; `claude-wayfinder-match` must be added as part of Deliverable 1.)

The following were deferred from the brainstorm to the implementation PR. Each is one sentence.

- **Duplicate-name collision behavior in `build_catalog.py`** — the spec calls for `project`-highest source-tagged precedence; verify `build_catalog.py`'s actual behavior before wiring the subcommand and align if it differs; deferred to implementation.

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| **Catalog drift** — consumer's built catalog becomes stale after agents/skills change, causing the matcher to route against an outdated picture | Medium | `docs/integration.md` documents refresh triggers; the Skill warns when catalog mtime is older than any source file (see Section 2.1 stale-mtime behavior) |
| **Schema evolution** — the catalog schema is in active migration: v6 sidecar schema supersedes v5 inline-frontmatter (see `build_catalog.py:L11-14`). `docs/schema.md` (Deliverable 4) is the declared stability contract for v0.2. If `docs/schema.md` is not produced before v0.2 ships, note explicitly in `docs/integration.md` that the catalog schema is mutable through v0.3, and commit to migration support as a v0.3 deliverable. | Medium | Produce `docs/schema.md` as Deliverable 4 of v0.2; schema stability is then asserted by that document |
| **Performance on large agent/skill trees** — `build_catalog.py` scans every source file; consumers with hundreds of skills may see slow builds | Low | v0.2 targets single-user harnesses, not large teams; punt to v0.3 if it surfaces |
| **Early-consumer contract stability** — power-users who adopt v0.2 at launch will write router prompts against the dispatch context JSON shape; breaking that shape in v0.3 has real adoption cost | Medium | Document the dispatch context shape in `docs/integration.md` as a versioned contract; any change requires a migration note in the release |
